# React Native New Architecture — Discipline

The New Architecture (Fabric renderer + TurboModules + JSI) is the default in RN 0.76 and the ONLY supported architecture in 0.82+. The legacy bridge is gone. Apply this checklist whenever you touch anything that crosses the JS / native boundary or affects render performance.

## 1. Never opt out

- Do NOT set `newArchEnabled: false` in `app.json` / `expo.json` / `Podfile`.
- Do NOT pin RN to a 0.81 or older version to "make the build pass."
- If a third-party lib forces an opt-out, escalate — pick a New-Arch-compatible replacement instead.

## 2. Forbidden legacy APIs (and their New replacements)

| Legacy (DO NOT use) | New replacement |
|---|---|
| `NativeModules.XYZ` direct property access | TurboModule registered via codegen; consume via the lib's typed export |
| `requireNativeComponent('Foo')` for a custom view | Fabric component generated by codegen from a TS spec |
| `Animated` API (`Animated.timing`, `Animated.Value`, `Animated.View`) | Reanimated 3 worklets (`useSharedValue`, `useAnimatedStyle`, `withTiming`) |
| `ListView` (and `DataSource`) | `FlashList` (or `FlatList` only when FlashList is overkill) |
| `AsyncStorage` from `react-native` core | `@react-native-async-storage/async-storage` or `react-native-mmkv` (with explicit task) |
| `NetInfo`, `Clipboard`, `ImagePickerIOS` from core | Their `@react-native-community/*` or Expo equivalents |
| Bridge-style `RCTEventEmitter` Obj-C / Java modules | TurboModule with `EventEmitter<T>` codegen spec |

If you see any of the left column in a file you are editing, fix it in the same change or surface it loudly (Rule 12).

## 3. TurboModule contract

When the project owns a TurboModule:

1. Define the JS spec as a TypeScript interface extending `TurboModule`, in `specs/Native<Name>.ts`.
2. Register it via codegen (`codegenConfig` in `package.json`); do NOT hand-roll the bridge file.
3. Methods are sync or `Promise`-returning. No `Callback` style (legacy bridge artifact).
4. Types must be JSI-supported primitives, objects, or arrays. No `Date`, no class instances — serialize at the boundary.
5. After changing a spec, re-run codegen (`npx expo prebuild --clean` or the lib's codegen script) before rebuilding native.

## 4. Fabric component lifecycle

When the project owns a Fabric component:

- Props spec lives next to the component as `<Name>NativeComponent.ts`, typed via `HostComponent<Props>`.
- Events are typed (`DirectEventHandler<T>` / `BubblingEventHandler<T>`) and named with the `on` prefix.
- Commands (imperative methods) go through `dispatchCommand` from `react-native/Libraries/ReactNative/RendererProxy`, not via refs returning native objects.
- Re-run codegen after any spec change; otherwise the native side will not see new props/events at runtime.

## 5. Performance rules under the New Architecture

- Memoize sparingly. With concurrent rendering, blanket `useMemo` / `useCallback` adds comparison overhead. Apply only to (a) heavy compute, (b) values passed to memoized children, (c) deps of effects in hot paths.
- Use FlashList for lists; ensure `estimatedItemSize` is set and recycling works (avoid per-item state that changes shape).
- Off-thread work goes into Reanimated worklets (`'worklet'` directive). The JS thread must stay free for input / scheduling.
- Prefer `useLayoutEffect` over `useEffect` only when the effect synchronously measures or mutates the DOM — otherwise stick to `useEffect` to let the renderer commit first.
- Profile with the Hermes profiler or Chrome DevTools (CDP via Hermes V1) before claiming a perf win. Numbers, not vibes.

## 6. Dependency compatibility check

Before adding any native dep:

1. Run `npx expo install <pkg>` (gets the SDK-compatible version, not the absolute latest).
2. Run `npx react-native-new-arch-check` — read the report; replace anything red.
3. For libs at the edge of compatibility, check the lib's GitHub for an open "New Architecture" tracking issue before committing.

## Review checklist

When reviewing a diff, fail the review on any of:

- Import of `Animated` from `react-native` (even via `Animated.View`).
- Direct `NativeModules.<Name>` property access.
- `requireNativeComponent` outside of a codegen-generated file.
- `newArchEnabled: false` introduced anywhere.
- A new native dep without a corresponding `react-native-new-arch-check` run.
- A worklet that closes over JS-thread state or React state setters without `runOnJS`.

Each of these is a defect under the New Architecture, not a style nit.
