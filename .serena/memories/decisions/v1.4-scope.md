## PO scope decision — v1.4

**Headline:** Refine all 13 profiles + add `react` and `react-native`. **Defer the handoff prompt to v1.4.1.** I take the CTO's pushback option, not their primary plan. Reasoning under section 5.

### MUST (ships in v1.4)

1. **Refine all 13 existing profile agents against the CTO Quality Bar (section 1).** Parallelizable, mechanical against a shared rubric, research bundle is fresh — staging causes drift.
   *Acceptance:* Each of the 13 profile agent .md files passes a 7-point rubric checklist (frontmatter minimal, description is "when," tools allowlisted, body has Role/When/Workflow/Output, no CLAUDE.md duplication, ≤2KB, single responsibility).

2. **Tighten existing skills in those 13 profiles to the Skill Bar.** No new skills, just sharpen what's there.
   *Acceptance:* Every existing `SKILL.md` opens with an unambiguous trigger sentence, is procedural (decision tree / checklist / commands), and cites at least one source rule from the v1.4 research bundle.

3. **Add `react` profile (Vite SPA, React 19, TanStack Query, Zustand, Tailwind 4, NativeWind-style, Vitest, React Router 7).**
   *Acceptance:* `npx ... --profile react` produces a working scaffold whose CLAUDE.md.partial, agent, commands, and skills all pass the same rubric as the refined 13.

4. **Add `react-native` profile (Expo managed, New Architecture, Expo Router, NativeWind, TanStack Form+Zod, Maestro, EAS Build).**
   *Acceptance:* `npx ... --profile react-native` produces a working scaffold meeting the same rubric, and the agent explicitly forbids editing Podfile/build.gradle without escalation.

5. **Wizard matrix update + smoke tests for both new profiles.**
   *Acceptance:* Wizard lists `react` and `react-native` under Developer; existing test suite runs against both with marker block and dogfood install asserts green.

### SHOULD (ships if MUST is on track)

6. **Single reviewer pass across all 15 refined/new profiles for cross-profile consistency.**
   *Acceptance:* One reviewer agent runs the rubric over all 15, files a diff or sign-off note per profile, zero rubric violations remain at merge.

7. **Update root README + CLI `--help` to surface the two new profiles and the refined-prompt note.**
   *Acceptance:* `npx @ennamjsc/agents-scaffold --help` lists 15 profiles; README install matrix shows react and react-native rows.

### COULD (parked for v1.5)

- Memory-field adoption on selected agents once we have 4+ weeks of usage data showing what's worth persisting.
- New skills beyond what each refined profile carries.
- Telemetry / opt-in install metrics (needs its own privacy decision first).

### WON'T (v1.4 — engineers do not slip these in)

- **Handoff prompt feature (CTO ask #3).** Moved to v1.4.1. See disagreement below.
- New MCP servers. serena + context7 + jira is the contract.
- Refactoring `_shared/` AGENTS.md rules or the Session Boot Protocol — out of scope; behavioral change risk on every existing install.
- Any change to the marker-block format itself.

### Disagreement with CTO — and my call

The CTO ranked the handoff prompt as highest risk and proposed mitigation (explicit "don't touch inside markers" + dry-run). They also offered a fallback: ship it standalone in v1.4.1. **I take the fallback.** Reason: the risk surface is a behavior we ask *another agent in another repo* to perform correctly, against the load-bearing marker-block API, with no automated test harness. Pairing it with 15 prompt rewrites means reviewer attention is split exactly where it needs to be undivided. v1.4.1 gets its own review cycle, a dedicated dry-run harness, and a real-world dogfood install before publish. The 15 refined/new prompts are valuable on their own and ship clean without it.

**Net v1.4 = bigger, safer surface area; v1.4.1 = the one risky feature, isolated.**

## Addendum — user override (2026-06-24)

The human user (project owner of record) reviewed the team's recommendation and **shipped the handoff prompt in v1.4** anyway, with the CTO's inline risk mitigations applied:

- Hard boundary rule in the prompt: "MUST NOT modify, reorder, or remove anything between those markers".
- Grounding rules: every claim must cite a file the agent actually read; unknown facts get `?` not a guess.
- Diff-before-write: prompt instructs the agent to show a unified diff and ask for confirmation before saving.
- CI suppression: handoff prompt is gated on `interactive` (suppressed under `--no-prompts`).

Implementation: `packages/cli/src/ux.ts` `printHandoffPrompt(profileName)` + wired in `index.ts` after `printNextSteps` + `maybeWarnStaleChromeDevtools`. Skipped for `profile === 'local-root'` (no app-shape to extract). Unit-tested in `tests/unit/handoff-prompt.test.ts`.

The PO defer rationale (split reviewer attention) remains a valid concern — it was traded against the user's preference to ship all three asks together. v1.4.1 is no longer earmarked for this feature.
