# Checkpoint: tech-lead — 2026-06-25 (v1.5.1)

## What was done

Patched two legacy `.claude/settings.json` shapes that the scaffold has been shipping since v1.0 and that current Claude Code rejects:

1. `permissions.additionalAllowList` → renamed to `permissions.allow` in the template. The legacy key was silently ignored — users' allow-rules for `Bash(npm:*)` etc. never took effect.
2. `hooks.SessionStart` bare `[{command}]` → wrapped in the required `[{hooks: [{type:"command", command}]}]` shape. The bare shape triggers Claude Code's "Expected array, but received undefined" parse-error popup.

Auto-rewrite is not possible because `mergeJson` is user-wins on arrays. Added `maybeWarnLegacySettings(cwd)` in `packages/cli/src/index.ts` that detects either legacy shape on install and warns loudly on stdout. Pattern mirrors the existing `maybeWarnStaleChromeDevtools`. Wired in after `maybeWarnStaleChromeDevtools` and before the handoff prompt.

Caught a logic bug during integration testing: the warner originally guarded `additionalAllowList` warning with `!('allow' in perms)`, which suppressed the warning whenever the merged result had BOTH keys (which is exactly what happens on a v1.5.0 → v1.5.1 re-run). Test caught it before ship; relaxed the condition.
